# Bybit Strategy Lab — актуальный README

## Что это за проект

Проект сравнивает расчётную торговлю стратегии и фактическое исполнение на Bybit.

Сейчас в проекте есть два главных контура:

1. **Расчёт стратегии**
   - стратегия пересчитывается по окну последних баров;
   - на графике и в таблице показываются сделки стратегии;
   - расчётная модель для стратегии сейчас использует `position_usd = 1000`.

2. **Реальное исполнение**
   - auto/manual/external события пишутся в журнал;
   - для реальных сделок собираются входы, выходы, комиссии и причины закрытия;
   - сравнение строится по `trade_key` / `entry_code`.

Проект развивался поэтапно, поэтому важно понимать: исторические старые данные могли быть записаны по старой логике, а новые сделки уже идут по обновлённой цепочке.

---

## Главное текущее правило

### Ключ сделки

Для auto-сделок используется **единый ключ сделки**:

`SYMBOL:TF:OPT_ID:ENTRY_TS`

Пример:

`APTUSDT:30:173:1774312200000`

Этот ключ:
- рассчитывается по **расчётному времени входа стратегии**, а не по времени фактического исполнения на бирже;
- используется как `order_link_id` / `entry_code`;
- участвует в сравнении strategy ↔ real.

Одна стратегия не означает один ключ.

Каждый новый вход — это **новая сделка** и **новый ключ**.
Если позиция закрылась по стопу и стратегия снова открылась в ту же сторону, это уже следующая сделка с новым `trade_key`.

## Текущая архитектура

### 1. Расчёт стратегии

Основной расчёт графика и backtest идёт через `_build_fig_and_bt(...)`.

Оттуда приходят:
- `fig`
- `bt`
- `err_html`

В `bt` используются:
- `bt.trades` — закрытые сделки стратегии;
- `bt.open_position` — текущая открытая позиция стратегии.

Важно:
- `bt.trades` сейчас не хранит исторические `trade_sl` / `trade_tp` в доступном виде;
- поэтому для старых strategy rows SL/TP нельзя честно достать из backtest-объекта задним числом.

### 2. Журнал стратегии

Для стратегических сделок используется таблица:

`strategy_trade_journal`

Она нужна, чтобы фиксировать то, что было у стратегии **в момент сделки**, а не пытаться восстановить это потом из плавающего окна.

Там могут храниться:
- `trade_key`
- `symbol`
- `tf`
- `strategy`
- `opt_id`
- `side`
- `entry_ts`
- `entry_price`
- `sl_price`
- `tp_price`
- `exit_ts`
- `exit_price`
- `close_reason`
- `status`
- `bybit_order_link_id`
- `created_ts`
- `updated_ts`

### 3. Реальные сделки

`real_trades` не хранится отдельной таблицей в БД.

Он собирается на лету из:
- `execution_events`
- `execution_fills`
- `execution_fill_summary`
- `exchange_snapshots`
- `live_events`

### 4. Вкладка `Сравнение`

Сравнение строится по `trade_key` / `entry_code`.

Для новых сделок логика уже рабочая: новые matched-строки появляются, и можно сравнивать цены входа/выхода и PnL.

### Что важно понимать

#### `Refresh strategy`

`Refresh strategy` сейчас должен использоваться **только вручную по кнопке**.

Он не должен запускаться автоматически по барам.

#### Автосинхронизация

При открытии `/trade/chart/data` может идти лёгкая синхронизация существующих journal-строк, но это не то же самое, что полный refresh.

#### Model size

В `Сделки стратегии` расчётный размер модели сейчас — **1000 USD**.

#### SL / TP в таблице стратегии

- `Model size` можно показывать честно;
- `SL / TP` для старых strategy rows не всегда можно взять из `bt.trades`, потому что backtest-объект их не хранит в доступном виде;
- для истории эти значения лучше брать из `strategy_trade_journal`, если они были сохранены в момент открытия сделки.

## Почему плавает стратегия и equity

Стратегия считается по **плавающему окну последних баров**.

Например: последние `2000` свечей.

Из-за этого:
- начало окна меняется;
- ранние сделки могут пересчитаться;
- суммарная прибыль может заметно отличаться;
- один и тот же текущий график может иметь похожую форму, но другой cumulative PnL.

Это нормально для такой модели.

### Следствие

Для точного сравнения с реальностью не стоит опираться только на общий cumulative PnL окна.

Надёжнее смотреть:
- конкретные сделки;
- конкретные `trade_key`;
- конкретные входы и выходы;
- конкретные причины закрытия.

---

## Что сейчас считать надёжным

### Надёжно
- новые real auto-сделки снова пишутся;
- новые `trade_key` для auto-сделок используются;
- comparison по новым сделкам уже даёт matched-строки;
- после стопа и повторного входа создаётся новый ключ, а не переиспользуется старый.

### Не полностью надёжно исторически
- старые строки, которые уже были перезаписаны/затёрты ранними refresh-логиками;
- исторический SL/TP, если он не был записан в момент открытия сделки;
- общий cumulative PnL на плавающем окне.

## Как правильно сравнивать сделки

Сравнивать надо **по одной сделке = по одному ключу**.

Для каждого `trade_key` смотреть:
- `entry_code`
- `strategy_entry_price`
- `real_entry_price`
- `strategy_exit_price`
- `real_exit_price`
- `strategy close_reason`
- `real close_reason`
- `strategy pnl`
- `real net pnl`

### Важная оговорка про размер позиции

Сейчас размеры разные:
- стратегия считает на `1000`;
- реальная торговля может идти на `25`.

Поэтому абсолютный USD-PnL напрямую сравнивать не всегда корректно.

Правильнее сравнивать:
- сторону;
- совпадение ключа;
- вход/выход в процентах;
- причину закрытия;
- нормализованный PnL.

## Журнал стратегии: что нельзя затирать

Для уже созданной исполненной строки identity-поля нужно считать неизменяемыми:
- `trade_key`
- `symbol`
- `tf`
- `strategy`
- `opt_id`
- `entry_ts`
- `side`
- `bybit_order_link_id`
- `created_ts`

Обновлять можно только состояние:
- `entry_price`
- `sl_price`
- `tp_price`
- `exit_ts`
- `exit_price`
- `close_reason`
- `status`
- `updated_ts`

И даже эти поля нельзя затирать `NULL`-ом, если уже есть сохранённое значение.

### Особо важно

Опасное место — логика вида `DELETE + INSERT` для среза журнала.

Если сделать `DELETE` раньше, чем перенесены исторические значения, можно потерять:
- `sl_price`
- `tp_price`
- `bybit_order_link_id`
- `created_ts`
- другие ранее зафиксированные данные.

## Как проверять новые изменения

Проверять новые изменения по журналу и сравнению нужно **только на новых auto-сделках**.

Порядок проверки:

1. появляется новая auto-сделка;
2. в `strategy_trade_journal` появляется новая строка;
3. у неё есть новый `trade_key`;
4. `sl_price` записывается и не затирается после refresh/sync;
5. после закрытия обновляются `exit_ts`, `exit_price`, `close_reason`, `status`;
6. во вкладке `Сравнение` появляется matched-строка.

## Текущее безопасное направление развития

### Уже можно делать
- смотреть новые matched-сделки во вкладке `Сравнение`;
- сравнивать входы/выходы и PnL по ключам;
- хранить historical `sl_price` в `strategy_trade_journal`;
- использовать `Refresh strategy` только вручную.

### Пока лучше не делать резко
- не включать авто-refresh по барам;
- не пересобирать журнал destructive-логикой `DELETE + INSERT` без переноса старых значений;
- не пытаться восстанавливать старые SL/TP из текущего плавающего backtest-запуска.
